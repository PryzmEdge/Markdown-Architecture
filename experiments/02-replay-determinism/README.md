# Experiment 02 — Replay Determinism

Tests ADR-005's purity requirement on the workflow layer. **This experiment is designed to fail on the current `proof/workflow.py`** — the failure is the productive result.

## Hypothesis

ADR-005 ("Temporal/DBOS Workflow-Activity Split for Durable Governance") requires the workflow layer to be deterministic and replayable:

> "Workflow layer (deterministic — must be replayable): Pure functions over event-log state. Validator gates, kill-switch logic, capacity checks, lattice policy enforcement... No direct I/O; no clocks; no RNG; no network calls. Must produce identical results when replayed from the same event log."

A replay of any complete run should re-derive bit-identical gate decisions AND a bit-identical receipt body. If anything diverges, the workflow violated ADR-005's purity contract.

`proof/explanation.md` § "The DBOS gap" already names this gap in prose: `workflow.py` calls `datetime.now(timezone.utc)` and `uuid.uuid4()` inside the workflow body. This experiment quantifies it.

## Method

`replay.py`:

1. Picks the most recent complete run from `provenance_log`.
2. Fetches its event log + recorded receipt.
3. Re-derives each gate's verdict from first principles: re-reads the input artifact, re-runs `ingester.assert_approved()`, compares against the recorded payload.
4. Rebuilds the receipt by calling `workflow.build_receipt()` with identical inputs (same `run_id`, same artifact, same operator, same `gate_check_results`).
5. Diffs the rebuilt receipt against the recorded receipt field-by-field.

Two verdicts:

- **Gate determinism** — are the recorded gate verdicts replayable?
- **Receipt determinism** — does the same `build_receipt()` call produce the same JSON?

Overall PASS requires both. The integrity anchor (`pg_receipt_hash = SHA-256(receipt_json)`) under replay is computed and reported.

## Results

Run against `workflow_run_id = ca3bebb3-...`. Full JSON: [`results/replay-results.json`](results/replay-results.json).

### Gate determinism: **PASS**

Recorded and replayed gate verdicts are bit-identical:

```json
{ "operator_approved": true, "status_approved": true, "contract_passed": true }
```

The gate is a pure predicate over the frontmatter (`assert_approved` reads `operator_approved` and `status`). Both sides produced the same answer. The gate layer satisfies ADR-005.

### Receipt determinism: **FAIL**

The recorded and replayed receipts produce different `SHA-256` hashes:

| | Recorded | Replayed |
|---|---|---|
| `pg_receipt_hash` | `b9d43b7e4af545e5...` | `5161d1ae3f907814...` |

10 fields match. 3 fields diverge:

| Field | Reason | Source |
|---|---|---|
| `receipt_id` | new UUID generated by `uuid.uuid4()` on each call | `proof/workflow.py:99` |
| `timestamp` | new ISO-8601 string from `datetime.now(timezone.utc)` on each call | `proof/workflow.py:97-100` |
| `operator_signoff` | nested object whose own `timestamp` is the same wall-clock value | `proof/workflow.py:115-119` |

### Overall: **FAIL** (as predicted)

The failure is exactly the failure `proof/explanation.md` § "The DBOS gap" warned about. ADR-005 requires the three offending calls to live inside activity wrappers (idempotency-keyed, retried, logged); right now they live in the workflow body.

## What the failure tells us

The receipt's body has **deterministic** content (the gate verdicts, the artifact hash, the YAML frontmatter snapshot, the agent ID, the cost ledger) interleaved with **non-deterministic** content (the receipt ID, the timestamps). A replay can't reproduce the receipt unless we either:

1. Move the non-deterministic bits into activities and record their results in the event log, so replay reads them rather than re-generates them; or
2. Pass them as inputs to `build_receipt()`, so a replay supplies the original values.

Option 1 is the DBOS-native fix. Option 2 is a smaller refactor that closes the same gap without adopting DBOS itself.

## The fix path (concrete)

To make this experiment **PASS** on the next iteration:

```python
# proof/workflow.py — minimal change to make build_receipt replayable
def build_receipt(run_id, artifact, operator_id, gate_results,
                  *, receipt_id, timestamp):  # NEW: required kwargs
    return {
        "receipt_id":   receipt_id,
        "timestamp":    timestamp,
        ...
        "operator_signoff": {
            "name":      operator_id,
            "timestamp": timestamp,  # share the workflow-level timestamp
            ...
        },
    }

# In run_workflow:
receipt_id = str(uuid.uuid4())          # was: inside build_receipt
timestamp = datetime.now(timezone.utc).isoformat().replace("+00:00", "Z")
log_event(conn, run_id, stage_id, "receipt_meta", "passed",
          {"receipt_id": receipt_id, "timestamp": timestamp})
conn.commit()

receipt = build_receipt(run_id, artifact, operator_id, gate_results,
                        receipt_id=receipt_id, timestamp=timestamp)
```

The non-deterministic values are generated once, logged as a `receipt_meta` event, and then consumed by `build_receipt()`. Replay reads the event payload and calls `build_receipt()` with the same values — receipts become bit-identical.

This is ~15 lines of change. Combined with the analogous fix for `run_id` (already-deterministic from the workflow caller's perspective), `workflow.py` would be fully ADR-005-compliant.

## What this proves and what it doesn't

**Proves:** ADR-005 is empirically falsifiable from the event log alone. The current `proof/workflow.py` violates the purity requirement on exactly the fields we expected (`receipt_id`, `timestamp`, and the nested `operator_signoff.timestamp`). No other receipt fields diverge — the violation is precisely localized.

**Does NOT prove:** That replay produces functionally-equivalent receipts. The recorded receipt has the actual `pg_receipt_hash` that was written to the integrity ledger; a replayed receipt with a different hash is, by construction, a different artifact. Replay doesn't "verify" the original receipt — it shows that a fresh receipt for the same logical run would not match.

**Does NOT prove:** That the fix above is the right fix. There are valid design alternatives — recording receipt metadata as activities, switching to DBOS proper, or treating the receipt as the *output* of a non-deterministic activity rather than something replay should reproduce. The choice is an ADR-worthy decision, not an obvious mechanical fix.
