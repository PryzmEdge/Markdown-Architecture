# Graphify in the Markdown Architecture Workspace

Graphify (`safishamsi/graphify`, PyPI: `graphifyy`) is an AST + community-clustering knowledge-graph tool for codebases. In this workspace it provides a token-efficient alternative to grep/find for codebase questions: a queryable graph of files, symbols, ADRs, and stage artifacts at `graphify-out/graph.json`.

This doc records **how Graphify is used in this workspace** — not what Graphify is. For the full reference, see the upstream repo.

---

## Install

```bash
pip install graphifyy       # installs the `graphify` CLI
graphify claude install     # writes ## graphify section to CLAUDE.md + PreToolUse hook
graphify update .           # build graphify-out/graph.json (AST only, no LLM/API cost)
```

The `claude install` step adds a `PreToolUse` hook on `Bash` to `.claude/settings.json`. The hook fires for grep/find/rg/fd/ack/ag commands when `graphify-out/graph.json` exists, injecting a one-line nudge that suggests `graphify query` instead. It is advisory, not blocking.

Web sessions are ephemeral — `pip install graphifyy` must be re-run when the container is recycled.

## Commands an agent will use

| Need | Command |
|---|---|
| "Find code/docs related to X" | `graphify query "X"` |
| "Explain this symbol/node" | `graphify explain "<label>"` |
| "How does A reach B?" | `graphify path "A" "B"` |
| "What's the overall shape?" | read `graphify-out/GRAPH_REPORT.md` |
| Rebuild after code edits | `graphify update .` |

The Python wrapper at `_config/skills/graphify_query.py` exposes `query`, `path`, and `explain` as importable functions for stage skills that need to call Graphify programmatically.

## What this workspace does **not** rely on

Two prior assumptions were dropped when v2 of `_config/skills/graphify_query.py` was written:

- **No HTTP MCP server on `:7331`.** Upstream Graphify is CLI + stdio, not HTTP. The earlier `graphify serve --config graphify.config.json` workflow was retired.
- **No frontmatter-aware helpers** (`get_blocked_artifacts`, `get_high_risk_artifacts`, `get_stage_slice`, `filter_nodes`). Upstream's default AST extraction does not lift YAML frontmatter onto graph nodes, so these never had a working backend. A separate skill that walks `stages/**/*.md` directly is the right home for those queries; it has not yet been written.

Semantic extraction (`graphify extract --backend ...`) can lift more metadata onto nodes but requires an LLM API key and incurs cost. Defer until needed.

---

## Interaction with workspace constraints

- **Required YAML frontmatter** on stage artifacts (`status`, `operator_approved`, `risk_check_passed`, `stage`) is enforced by `_config/stage-contract.py`, not by Graphify. Graphify sees these files but does not validate them.
- **`operator_approved` gate** for stage promotion is enforced by hooks in `.claude/settings.json`, not by Graphify.
- **`graphify-out/`** is gitignored. CI rebuilds it on demand; commits do not carry generated graph artifacts.
- **CLAUDE.md token budget (≤800).** Upstream's `## graphify` section added ~120 words and currently sits within budget. If CLAUDE.md grows, trim the upstream section first since this doc carries the long form.
