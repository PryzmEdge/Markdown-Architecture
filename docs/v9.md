## Executive Summary

Markdown is a **human-readable orchestration substrate** — not an execution environment. Its consumers (parsers, runtimes, LLM agents, Pandoc filters, notebook kernels) do the executing. The emerging production architecture is best understood as **deterministic scaffolding around probabilistic reasoning cores**: a pattern where deterministic filesystems, schemas, hooks, and pipelines wrap a nondeterministic LLM reasoner. This is the long-term winning pattern for quant workflows — and for any domain where provenance, reproducibility, auditability, and staged human approval are non-negotiable.

> **Core architectural claim, version 4 (calibrated):**
> *"The LLM is not the system. It is a probabilistic reasoning subsystem inside a deterministic operating environment. Orchestration, state, and tooling quality determine production reliability — not model capability alone."*

*(NIST AI RMF GOVERN-1.2)*

An **operator** is a named person or named team who:
1. Has read and understood `domain-rules.md` and `execution-doctrine.md`
2. Has access credentials to the audit log and `execution_receipts/`
3. Is authorized to approve stage promotions (AMBER gates) and reject RED gate outputs

### Artifact State vs. Execution State

| State Type      | Where It Lives                         | Example                                    |
|----------------|----------------------------------------|--------------------------------------------|
| Artifact state  | Filesystem (markdown, YAML, outputs)   | Stage CONTEXT.md, analysis-config.yaml     |
| Execution state | Agent runtime (transient)              | In-flight tool calls, token attention, reasoning chain |

> *"ICM externalizes durable workflow state into human-readable artifacts. Transient execution state still partially exists within the agent runtime."*

This distinction matters for replay, debugging, and compliance: Git captures artifact state; it does not capture execution state. For complete execution-state capture, use `PromptExecutionReceipt` (Part 11).

## Part 3 — Deterministic Scaffolding Around Probabilistic Cores

This is the dominant production architecture pattern in 2026.

From "Harness Engineering: The Architecture of Production-Grade AI Systems" (dev.to, May 2026):

> *"The architectural fix is to treat the LLM as a probabilistic policy inside a deterministic harness. By inserting deterministic steps — rules, finite state machines, and hard-coded policies — into the workflow, engineers can halt the compounding loss of reliability."*

The math is unforgiving: a 10-step agent chain at 95% per-step accuracy yields **59.9% end-to-end reliability** (0.95^10). One deterministic circuit breaker inserted at a critical step resets reliability for that checkpoint to 100%.

> **Calibration on the LLM analogy:** The correct framing is **"LLM as stochastic policy with tool-call side effects"** — not "LLM as CPU." A CPU executing the same instruction sequence twice produces bit-identical state transitions; an LLM does not, even at temperature 0 with fixed seed (kernel non-determinism, batching effects, vendor-side routing). Anthropic's Opus 4.7 ships a new tokenizer that produces up to 35% more tokens for the same input as Opus 4.6 — meaning token accounting is not stable across model versions. This non-determinism matters for audit reconstruction and compliance reproducibility (see NIST AI RMF MEASURE 2.5, ISO 42001 Clause 9).

### Compiled AI: Zero-Token Deterministic Execution

*(arXiv:2604.05150)*

A primary mechanism to achieve 100% execution-stage determinism is **Compiled AI**. LLMs are leveraged exclusively during a compile-time generation phase to emit static, executable code artifacts. Once generated, these workflows execute deterministically within pre-validated templates on conventional software infrastructure, completely eliminating runtime model calls.

Compiled AI satisfies three core systems properties:

1. **One-time LLM invocation:** The model runs once at generation time, not at transaction time.
2. **Zero-token deterministic execution:** Deployed workflows execute as native code (Python, Terraform) without further API calls — measured transaction-stage latency: 4.5ms (450× improvement over direct LLM runtimes).
3. **Mandatory multi-stage validation:** Generated code is subjected to automated syntax, execution, and semantic accuracy checks before deployment, converting runtime failures into detectable compile-time blocks.

### The Three Enforcement Layers

| Enforcement Type              | Mechanism                               | Reliability                            |
|------------------------------|-----------------------------------------|----------------------------------------|
| CLAUDE.md instructions        | Advisory context (user-message tier)    | Probabilistic                          |
| Schemas (Pydantic/JSON Schema)| Structured output constraints           | High — rejects invalid outputs         |
| Hooks (.claude/settings.json) | PreToolUse/PostToolUse callbacks        | Deterministic — guaranteed execution   |

**Production rule:** Never rely on CLAUDE.md for hard constraints in quant workflows. Behavioral rules belong in hooks. Schema contracts belong in Pydantic models (v2, pinned ≥2.13.4). CLAUDE.md handles structural navigation and identity only.


## Part 4 — The Memory Hierarchy: LLM Context as Cost-Tier Cache

*(arXiv:2603.09023 — "The Missing Memory Hierarchy: Demand Paging for LLM Context Windows," Tony Mason, March 2026)*

Abstract, verbatim:
> *"The context window of a large language model is not memory. It is L1 cache: a small, fast, expensive resource that the field treats as the entire memory system. There is no L2, no virtual memory, no paging. Every tool definition, every system prompt, and every stale tool result occupies context for the lifetime of the session."*

Key measurement: across 857 production sessions and 4.45 million effective input tokens, **21.8% was structural waste** (stale tool results, redundant definitions, dead context). The Pichay demand-paging system reduced context consumption by up to 93% in production.

> **Framing note:** This hierarchy is a **cost-tier** model, not a hardware-cache model. There is no hardware-enforced coherence between layers. The "fill cost" varies by an order of magnitude depending on the TTL chosen (Anthropic prompt cache: 5-minute default, 1-hour optional; cache hits at 10% of standard input price; cache writes at 1.25×–2× base input). Eviction policies are heuristic, not LRU.

### The Full Memory Hierarchy

| Level                  | OS Analogy           | Context Content                                       | Persistence      | $/MTok reference       | Source                |
|------------------------|----------------------|-------------------------------------------------------|------------------|------------------------|-----------------------|
| L1 — Generation Window | CPU L1 cache         | Active turn context, system prompts, current tokens   | Turn lifetime    | Full input rate        | arXiv:2603.09023      |
| L2 — Working Set       | CPU L2 cache / RAM   | Pinned, high-fault-history context & tool state       | Session lifetime | Cached input (10%)     | arXiv:2603.09023      |
| L3 — Session History   | Swap partition       | Compressed/compacted turns with declared losses       | Session lifetime | Negligible (disk)      | arXiv:2603.09023      |
| L4 — Persistent Memory | Local Storage        | External, cross-session persistent knowledge          | Permanent        | Negligible (disk)      | arXiv:2603.09023      |
| L5 — Git History †     | Archival             | Diffs, version state at any commit                    | Permanent        | Negligible (disk)      | **Extended by author** |

> † **Author extension.** arXiv:2603.09023 defines L1–L4. L5 (Git history as explicit archival layer) is an extension applied here for quant/compliance workflows where prompt provenance and artifact versioning must be independently auditable. It is not present in Mason's original hierarchy.

### Pichay: Virtual Memory Demand Paging for LLMs

Pichay interposes as a transparent proxy between the client application and the LLM inference endpoint, actively managing context tokens through three mechanisms:

- **Context Eviction:** Stale files or past tool outputs are surgically evicted from L1. A lightweight retrieval handle (e.g., `<page_ref id="file_x">`) is written in their place — a virtual page table entry.
- **Page Fault Detection:** If downstream reasoning references the paged-out file, the proxy intercepts, registers a "page fault," and triggers a tool call to page the raw data back into L1.
- **Inverted Cost Model:** For LLMs, keeping active tokens in context is expensive (quadratic self-attention scaling); re-fetching is a cheap, one-time transaction cost. Aggressive eviction of cold context is mathematically and financially optimal. This is the opposite of CPU cache economics.

### Context Performance Claim (calibrated)

> *"Large monolithic prompts reduce instruction fidelity and retrieval precision. Research shows critical information in the middle of long contexts showed accuracy degradation in 2023-era models (Liu et al., TACL 2024 — 'Lost in the Middle', DOI 10.1162/tacl\_a\_00638) — frontier models (Gemini 2.5 Flash) have substantially mitigated this (McKinnon, Google LLC, arXiv:2511.05850, 2025: 26/26 questions answered correctly at every context-size step). Blanket 'LLMs lose middle context' claims are not universally valid for 2025–2026 frontier models."*

**Correct metric names** (Liu et al.): *multi-document QA accuracy* and *key-value retrieval accuracy* on synthetic UUID lookups — **not F1, NDCG, or "Number Match."**

From Anthropic's Claude Code best practices (code.claude.com/docs/en/best-practices, May 2026):
- CLAUDE.md is read at session start and on every turn
- Content is delivered as a **user-message after the system prompt** (not forced system config)

- Recommendation: "Only include things that apply broadly. For domain-specific workflows, use skills."

> *"Claude Code maintains CLAUDE.md as persistent contextual guidance throughout the session. The content is advisory — Claude applies relevance judgment. For guaranteed execution, use hooks."*

### Interactive TUI Context Commands

```bash
# Inspect current context window usage
/context-window

# Manual compact — lossy summary, resets window
/compact

# Clear specific tool results inline
/clear-tool-results
```

**Context anxiety:** Empirical testing shows frontier models (Claude 3.5/4) may wrap up tasks prematurely or truncate logical steps when sensing context limit approach. Mitigations:

- **Compaction:** Lossy, whole-transcript operation that flattens conversation history into a condensed summary, resetting the context window while preserving high-fidelity structural facts.
- **Tool-Result Clearing:** Precise, sub-transcript operation that replaces massive `tool_result` content blocks (raw code readouts, database dumps) with lightweight placeholders, preserving the original `tool_use` signature to maintain reasoning continuity while reclaiming up to 90% of active token space.

| Behavioral     | "Never touch production", "Always use env vars" | Probabilistic | Use hooks      |

## Part 8 — MCP: The Tool-Invocation Transport (Correctly Framed)

### Architecture Analogy

| Component     | OS Analogy                  | Role                              |
|--------------|-----------------------------|-----------------------------------|
| MCP protocol  | Syscall ABI                 | Standardizes invocation semantics |
| MCP tools     | Device drivers              | Execute against external systems  |
| Tool schemas  | Interface contracts         | Define input/output types         |
| Agent runtime | Scheduler / process manager | Decides when to invoke            |

MCP itself does not execute — it **standardizes invocation semantics**.

> ⚠️ **Critical caveat on the syscall analogy:** True syscall architectures enforce privilege rings, kernel mediation, capability enforcement, and hardware memory boundaries. **Current MCP implementations have none of these.** MCP is fundamentally trust-on-first-use, prompt-mediated, and text-defined. The MCP specification explicitly states: *"Tools represent arbitrary code execution and must be treated with appropriate caution. In particular, descriptions of tool behavior such as annotations should be considered untrusted, unless obtained from a trusted server."* (modelcontextprotocol.io/specification/2025-11-25)
>
> The syscall analogy is useful for **reasoning about layering** — it is **not a security claim**. For any production system with access to sensitive data or external execution, treat MCP tool responses as **untrusted input** requiring explicit validation.

### MCP Threat Model (required for quant)

*(arXiv:2504.03767, MCP Safety Audit, 2025)*

| Threat Vector | Description | Mitigation |
|--------------|-------------|------------|
| Prompt injection | Tool outputs inject adversarial instructions | Validate every tool response before using in execution decisions |
| Tool shadowing / server impersonation | Malicious server mimics legitimate tool | Pin server URLs; verify tool schemas at session start |
| Overprivileged tokens | OAuth tokens grant broader access than needed | Least-privilege token scoping per tool |
| JSON-RPC denial-of-wallet loops | Runaway tool calls consume token budget | Rate limits in AI Gateway (Part 22); budget caps per session |
| Unauthorized data exfiltration | Malicious server responses exfiltrate context | Output validation; network egress controls |

> **OWASP LLM Top 10 reference:** LLM10 — Unbounded Consumption applies directly to denial-of-wallet scenarios.

**Hard rule for your quant workspace:** Validate every MCP tool response before using it in an execution decision. Log every MCP tool invocation as part of `PromptExecutionReceipt` (Part 11).

### 2026 MCP Roadmap Status

Active development on MCP Server Cards (`.well-known` discovery), scalable session handling, conformance test suites. **No conformance testing exists yet.** Treat as a rapidly evolving protocol, not a stable standard. (modelcontextprotocol.io/development/roadmap)

### Production Retrieval Library Recommendation

> ⚠️ **`rank-bm25` is pedagogy-only.** Version 0.2.2 (last release 2022). > Additionally, `rank-bm25` does no preprocessing (no lowercasing, no stopword removal, no stemming) — pathological scores on real documents without explicit preprocessing.

**Production recommendations:**
- **In-process:** [`bm25s`](https://pypi.org/project/bm25s/) (Xing Han Lu, 2024) — scipy-sparse-matrix backed, ships multilingual stopword list and stemmer hooks, orders-of-magnitude faster
- **Production scale:** Elasticsearch or OpenSearch

## Part 10 — Frontier Concept A: Typed Orchestration / Schema-Constrained Output

This is the "static typing for AI orchestration" pattern — one of the most important production shifts in 2026.

### The Core Insight (arXiv:2410.18146, Meaning Typed Prompting, 2024)

Rigid JSON schemas alone reduce LLM reasoning quality. The superior approach: **Pydantic field descriptions become instructions to the model** — the schema IS the prompt.

> **All Pydantic code in v9 uses v2 patterns (pinned ≥2.13.4).**
> Migration: `@validator` → `@field_validator` + `@classmethod`; `class Config:` → `model_config = ConfigDict(...)`; `.dict()` → `.model_dump()`; `.json()` → `.model_dump_json()`; `parse_raw` → `model_validate_json`.

```python
from pydantic import BaseModel, Field, field_validator, ConfigDict
from typing import Literal
import instructor
import anthropic

client = instructor.from_anthropic(anthropic.Anthropic())

class StageOutput(BaseModel):
    """Structured stage evaluation output — the schema IS the prompt."""
    model_config = ConfigDict(str_strip_whitespace=True)

    artifact_id: str = Field(description="Unique identifier for this output artifact, kebab-case")
    status: Literal["pass", "fail", "needs-review"] = Field(
        description="Validation result. Only 'pass' if all acceptance criteria are met."
    )
    confidence: float = Field(
        ge=0.0, le=1.0,
        description="Confidence score 0.0–1.0. Below 0.7 defaults to 'needs-review'."
    )
    constraint_status: Literal["GREEN", "AMBER", "RED"] = Field(
        description="RED means block promotion. AMBER requires operator confirmation."
    )
    rationale: str = Field(
        max_length=200,
        description="One-sentence justification referencing the specific constraint checked."
    )
    operator_approval_required: bool = Field(
        description="True if constraint_status is AMBER or RED, or confidence < 0.85."
    )

    @field_validator("confidence")
    @classmethod
    def confidence_implies_review(cls, v, info):
        # Handled by operator_approval_required field — validation is informational
        return v

result = client.chat.completions.create(
    model="claude-opus-4-5",
    response_model=StageOutput,
    messages=[{"role": "user", "content": f"Evaluate stage output for {artifact_id}. Context: {context}"}]
)

if result.constraint_status == "RED" or result.operator_approval_required:
    raise ValueError(f"Operator approval required: {result.rationale}")
```

### Stage Contract as Typed Schema (Pydantic v2)

```python
from pydantic import BaseModel, ConfigDict
from typing import Literal
from datetime import date

class StageContract(BaseModel):
    model_config = ConfigDict(str_strip_whitespace=True)

    stage_id: str
    status: Literal["draft", "ready", "in-review", "approved", "published"]
    operator_approved: bool = False
    publish_target_date: date | None = None
    risk_check_passed: bool = False
    inputs_verified: bool = False

    @property
    def can_proceed_to_next_stage(self) -> bool:
        return (
            self.status == "approved"
            and self.operator_approved
            and self.risk_check_passed
            and self.inputs_verified
        )
```

## Part 11 — Frontier Concept B: Prompt Provenance & AI Audit Logging

### Why Git Is Insufficient

Git captures artifact state (file content at each commit). It does not capture:
- Which prompt generated which output
- Which model version / tokenizer version was active
- What context was loaded at inference time
- Which tool calls were made with what arguments and what responses were returned
- What MCP server versions were connected
- What it cost — per token, per stage, per workflow

For regulated quant/trading environments, this gap is a compliance liability.

### Regulatory Landscape (2026)

- **EU AI Act Art. 12(1)–(2)** (Regulation (EU) 2024/1689): High-risk AI systems must technically allow automatic recording of events over the system lifetime. Art. 12(2)(a) requires logs identifying situations that may present a risk. Art. 26(6): deployers retain logs minimum 6 months.
- **HIPAA §164.312(b)** (45 CFR 164.312): Audit controls required when systems handle ePHI — whether AI prompts constitute PHI is context-dependent; err on the side of inclusion.
- **ISO 42001:2023 §7.5**: Records of AI system decisions must be maintained. Specify retention period and storage class.
- **Sector-specific:** Financial regulators (SEC, FCA) are publishing AI governance guidance; consult current guidance for your jurisdiction.

> **Receipt retention policy:** Receipts must be retained for a minimum of **6 months** (EU AI Act Art. 26(6)) and **6 years** for FINRA-regulated workflows. Storage class: WORM (Write Once Read Many) or equivalent append-only immutable store. Never allow post-hoc mutation.

### Production Prompt Provenance Schema (Pydantic v2, extended)

```python
import hashlib
import uuid
from datetime import datetime, timezone
from pydantic import BaseModel, ConfigDict

class CostFields(BaseModel):
    tokens_in_uncached: int = 0
    tokens_in_cached: int = 0
    tokens_out: int = 0
    tokens_thinking: int = 0
    usd_estimated: float = 0.0
    usd_currency: str = "USD"
    model_pricing_version: str = ""
    cache_hit_ratio: float = 0.0

class BudgetFields(BaseModel):
    stage_id: str = ""
    stage_budget_usd: float = 0.0
    cumulative_workflow_usd: float = 0.0

class PromptExecutionReceipt(BaseModel):
    model_config = ConfigDict(str_strip_whitespace=True)

    receipt_id: str           # UUID
    timestamp: str            # ISO 8601 UTC
    model_id: str             # "claude-opus-4-5"
    model_version: str        # API-reported version string
    tokenizer_version: str    # Track across Opus 4.6 → 4.7 migration
    inference_geo: str        # "us" / "eu" — affects pricing 1.1×
    endpoint_url: str         # Exact endpoint for replay
    cache_hit: bool = False
    prompt_cache_ttl_seconds: int = 300
    retry_count: int = 0
    upstream_error_codes: list[str] = []
    prompt_hash: str          # SHA-256 of full prompt text
    context_manifest: list[str]   # File paths + git SHAs loaded
    tool_invocations: list[dict]  # MCP tool calls with args + response hashes
    output_hash: str          # SHA-256 of response text
    operator_id: str | None   # Who approved, if approval required
    stage_id: str
    latency_ms: int = 0
    provider_request_id: str = ""
    cost: CostFields = CostFields()
    budget: BudgetFields = BudgetFields()

def create_receipt(
    model_id: str,
    prompt: str,
    context_files: list[str],
    output: str,
    tool_calls: list[dict],
    stage_id: str,
    model_version: str = "",
    tokenizer_version: str = "",
    inference_geo: str = "us",
    endpoint_url: str = "",
    cache_hit: bool = False,
) -> PromptExecutionReceipt:
    return PromptExecutionReceipt(
        receipt_id=str(uuid.uuid4()),
        timestamp=datetime.now(timezone.utc).isoformat(),
        model_id=model_id,
        model_version=model_version,
        tokenizer_version=tokenizer_version,
        inference_geo=inference_geo,
        endpoint_url=endpoint_url,
        cache_hit=cache_hit,
        prompt_hash=hashlib.sha256(prompt.encode()).hexdigest(),
        context_manifest=context_files,
        tool_invocations=tool_calls,
        output_hash=hashlib.sha256(output.encode()).hexdigest(),
        operator_id=None,
        stage_id=stage_id,
    )
```

> **Signing recommendation:** Sign receipts with Ed25519 (not just SHA-256) for repudiation resistance, not merely tamper-evidence. Persist in an append-only store. Run daily reconciliation against Anthropic's billing CSV to detect cost anomalies.
